## 1. 节点配置错误

### ❌ 错误：节点 label 没有意义
```
label: "Agent 1"
label: "研究 Agent"
label: "处理节点"
```
**正确做法：** label 应该是具体的角色名，让人一看就知道它做什么
```
label: "竞品研究员"
label: "财务数据分析师"
label: "用户体验评估师"
```

---

### ❌ 错误：systemPrompt 太简单
```
systemPrompt: "你是一个研究助手，请帮助用户研究相关信息。"
```
**问题：** 太模糊，Agent 不知道该怎么工作，输出质量低且不稳定。

**正确做法：** 包含角色定位、具体职责、工作方式、输出要求四个部分，最少 150 字。

---

### ❌ 错误：maxIterations 设置不合理
```
// 简单总结任务设置 20 次迭代 → 浪费费用
maxIterations: 20

// 复杂深度研究任务设置 3 次迭代 → 结果太浅
maxIterations: 3
```
**正确做法：**
- 简单任务（格式转换、总结）：5-8
- 中等任务（分析、研究）：10-15
- 复杂任务（深度研究、完整报告）：15-25

---

### ❌ 错误：completionCriteria 不具体
```
completionCriteria: ["完成任务", "输出结果"]
```
**正确做法：** 具体可验证的标准
```
completionCriteria: [
  "报告包含至少5个竞品的分析",
  "每个竞品都有真实用户评价数据",
  "最终报告长度在2000字以上"
]
```

## 2. 结构设计错误

### ❌ 错误：本可以并行的任务强行串行
```
研究员A → 研究员B → 研究员C → 分析师
```
**问题：** 三个研究员互相独立，没有依赖关系，串行执行浪费时间。

**正确做法：**
```
研究员A ──→┐
研究员B ──→ Merge → 分析师
研究员C ──→┘
```

---

### ❌ 错误：多路并行后直接连一个 Agent
```
研究员A ──→┐
研究员B ──→ 分析师  ← 错误！
研究员C ──→┘
```
**问题：** 多路输入连到一个 Agent 时，该 Agent 只会接收到最后一个输入，前面的输入会丢失。

**正确做法：** 必须加 Merge 节点
```
研究员A ──→┐
研究员B ──→ Merge → 分析师
研究员C ──→┘
```

---

### ❌ 错误：没有质量门控就直接输出
```
执行者 → Output
```
**问题：** 对于重要的输出，没有质量检查，结果可能不达标。

**正确做法：** 重要输出前加质检
```
执行者 → 质检员 → Condition ──pass──→ Output
                      └──fail──→ 执行者（重做）
```

---

### ❌ 错误：循环没有退出条件
```
// Condition 的 false 分支连回执行者，但没有设置 maxLoopIterations
```
**问题：** 如果执行者一直无法通过质检，会无限循环。

**正确做法：** 在 Condition 节点配置 maxLoopIterations，建议设 2-3

---

### ❌ 错误：节点太少，职责混乱
```
Input → 万能 Agent（研究+分析+写作+质检） → Output
```
**问题：** 一个 Agent 承担太多职责，context 过长，容易出错，输出质量不稳定。

**正确做法：** 拆分成职责单一的多个 Agent

---

### ❌ 错误：节点太多，过度设计
```
Input → 理解需求 → 拆解任务 → 搜索A → 搜索B → 搜索C
→ 整合搜索 → 初步分析 → 深度分析 → 初稿 → 修改稿 → 润色 → 质检 → Output
```
**问题：** 节点过多，协调成本高，费用高，不必要的复杂度。

**正确做法：** 能合并的合并，保持 flow 简洁。通常 4-8 个节点就够了。

## 3. systemPrompt 写法错误

### ❌ 错误：只说"是什么"，不说"怎么做"
```
你是一位市场研究专家，负责研究竞品信息。
```

**正确做法：** 说明具体工作方式
```
你是一位市场研究专家...
工作方式：
- 先搜索官网了解产品定位
- 再搜索用户评价（G2、Capterra、Reddit）了解真实体验
- 区分官方宣传和用户真实反馈
- 每个竞品至少搜索 3 个以上来源
```

---

### ❌ 错误：输出要求模糊
```
请输出一份详细的分析报告。
```

**正确做法：** 具体说明格式、内容、长度
```
输出要求：
- 使用 Markdown 格式
- 结构：执行摘要（150字）→ 各竞品分析 → 对比表格 → 结论
- 每个竞品分析 300-500 字
- 最终总长度 2000-3500 字
- 关键数据加粗突出
```

---

### ❌ 错误：对质检节点的输出格式要求不清晰
```
请评估上游输出的质量，如果好就通过，不好就拒绝。
```

**问题：** Condition 节点需要解析质检员的输出，格式不规范会导致判断失败。

**正确做法：** 严格要求 JSON 输出
```
你必须且只能输出以下 JSON 格式，不要有任何其他内容：
{
  "passed": true 或 false,
  "score": 1到10的整数,
  "feedback": "改进建议"
}
```

## 4. 工具配置错误

### ❌ 错误：所有节点都开 web_search
```
// 写作助手、格式整理、内容总结等节点也开了 web_search
enabledTools: ["web_search", "url_fetch", "code_execute"]
```
**问题：** 工具越多，Agent 越容易在不必要的时候调用工具，浪费 token 和时间。

**正确做法：** 只给真正需要的节点开工具
- 需要联网查信息 → web_search
- 已知 URL 需要读取 → url_fetch
- 需要执行代码 → code_execute / python_execute
- 纯文本处理 → 不开任何工具

---

### ❌ 错误：需要搜索的节点不开工具
```
// 竞品研究员居然没有 web_search
{
  "label": "竞品研究员",
  "enabledTools": []
}
```
**问题：** 没有工具的研究员只能靠训练数据，信息严重过时。

**正确做法：** 所有需要获取最新信息的节点都要开 web_search。

## 5. 基础结构错误（最常见）

### ❌ 错误：没有 Input 节点
```json
// flow 里没有 type: "io" 的节点
// 执行引擎找不到起点，flow 无法运行
```
**正确做法：** 每个 flow 必须有且只有一个 Input 节点。

---

### ❌ 错误：没有 Output 节点
```json
// flow 里没有 type: "output" 的节点
// 执行完没有地方展示结果
```
**正确做法：** 每个 flow 必须有至少一个 Output 节点。

---

### ❌ 错误：有节点但没有 edges
```json
{
  "nodes": [input, agent1, agent2, output],
  "edges": []   // 空的！
}
```
**问题：** 节点之间没有连线，执行引擎不知道执行顺序，所有节点都不会执行。

**正确做法：** 每个节点都必须通过 edges 连接。
- input → agent（第一个执行的节点）
- agent → agent（串行）或 agent → merge（并行汇总）
- 最后一个 agent → output

---

### ❌ 错误：flow 没有 name
```json
{
  "id": "flow-123",
  "nodes": [...],
  "edges": [...]
  // 没有 name 字段
}
```
**正确做法：** 每个 flow 必须有有意义的 name 字段。

---

### ❌ 错误：孤立节点（有节点无连线）
```json
// 图中能看到节点，但 edges 里没有连接它的记录
// 用户截图反馈："节点在画布上但不执行"
```
**正确做法：** 生成完 JSON 后自查：
1. 每个节点 id 是否都出现在至少一条 edge 的 source 或 target 中？
2. Input 节点 id 是否在某条 edge 的 source 中？
3. Output 节点 id 是否在某条 edge 的 target 中？
